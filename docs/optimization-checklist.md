# 四宫格九宫格拆分工具优化清单

本文档整理当前项目可优化事项，按优先级分为稳定性、安全性、体验、结构和扩展功能五类。

## 一、优先级最高

### 1. 统一真实入口

- 当前 `.spec` 打包入口是 `gui_splitter_plus.py`。
- `gui_splitter_plus_copy.py` 包含额外的“镜头表生成”功能，但没有被当前打包配置使用。
- 建议决定正式主线：
  - 只保留图片拆分工具：归档或删除 `gui_splitter_plus_copy.py`。
  - 保留镜头表功能：合并到正式入口，并更新 `.spec`。
  - 将正式入口改名为 `main.py` 或 `app.py`，减少歧义。

### 2. 删除功能默认进回收站

- 当前 `gui_splitter_plus.py` 中默认永久删除：`use_trash_default = False`。
- 建议默认改为进回收站：`use_trash_default = True`。
- 如果未安装 `send2trash`，应明确提示安装，或永久删除前增加更强确认。

### 3. 增加依赖文件

- 新增 `requirements.txt`，记录运行和打包依赖。
- 主版本最少依赖：
  - `Pillow`
  - `send2trash`
- 如果保留 LLM 镜头表功能，还需要：
  - `requests`

### 4. 修复或归档旧版本文件

- `gui_splitter.py` 中 `ttk.Combobox` 使用了无效参数 `earlier`，运行可能报错。
- 建议：
  - 修复该参数；或
  - 移入 `legacy/`；或
  - 删除旧版，只保留正式入口。

### 5. 增强删除目录防误操作

- 删除面板允许用户选择任意目录并递归删除图片，存在误删风险。
- 建议增加保护：
  - 禁止选择磁盘根目录、用户主目录等高风险路径。
  - “删除全部”前展示待删除数量和部分文件名。
  - 待删除数量较大时要求输入确认文本。
  - 默认只允许删除输出目录内图片，除非用户显式解锁。

## 二、功能体验优化

### 6. 拆分前显示预计输出数量

- 批量拆分前显示输入图片数量和预计输出切片数量。
- 示例：`将处理 12 张输入图片，预计输出 48 张切片`。

### 7. 自动判断网格更可解释

- 当前自动判断只根据宽高比判断 `2x2` 或 `3x3`。
- 建议状态栏展示判断依据。
- 示例：`自动判断为 2x2，原因：宽高比 0.56`。
- 对不确定区间可提示用户手动确认。

### 8. 批量输出支持保留子目录结构

- 当前批量处理会把所有输出放到同一个目录。
- 建议增加选项：
  - 扁平输出：当前行为。
  - 保留目录结构：按输入文件夹结构输出切片。

### 9. 打开输出目录前自动创建目录

- 打开输出目录时，如果目录不存在，建议统一自动创建。
- 创建失败时再提示错误。

### 10. 输出格式说明更准确

- 当前界面强调“无损裁切”。
- 但如果选择 `keep` 且原图是 JPG，保存时仍会重新编码，不是严格无损。
- 建议文案改为：
  - `png/webp：无损保存`
  - `keep：尽量保持原格式，JPG 会重新编码`

## 三、代码结构优化

### 11. 拆分核心逻辑和 GUI

- 当前 `gui_splitter_plus.py` 文件较大，GUI、裁切、保存、删除逻辑混在一起。
- 建议拆分为：
  - `main.py`：程序入口。
  - `image_splitter.py`：图片判断、网格判断、裁切和保存。
  - `file_ops.py`：打开目录、删除文件、回收站处理。
  - `splitter_gui.py`：图片拆分 GUI。
  - `shot_table.py`：镜头表生成逻辑，如果保留该功能。

### 12. 为核心裁切逻辑增加测试

- 建议测试场景：
  - `1000x1000` 拆 `2x2`，得到 4 张 `500x500`。
  - `1001x1001` 拆 `2x2`，总覆盖范围不丢像素。
  - `1000x999` 拆 `3x3`，所有边界覆盖完整图片。
  - 输出文件重名时生成 `_001`、`_002` 等后缀。

### 13. Worker 线程不要直接读取 Tk 变量

- 当前后台线程中直接读取 `self.grid_mode.get()`、`self.out_format.get()`。
- Tkinter 状态变量不应跨线程直接访问。
- 建议在线程启动前读取配置快照，再作为参数传入 worker。

### 14. 统一日志符号

- 当前日志中有 `✅`、`❌`、`⚠️` 等符号。
- 如果考虑 Windows 字体和打包环境兼容性，可改为 ASCII：
  - `[OK]`
  - `[FAIL]`
  - `[WARN]`
  - `[DONE]`

### 15. 增加 `.gitignore`

- 建议忽略构建产物和输出目录：

```gitignore
build/
dist/
__pycache__/
*.pyc
output_tiles/
shots_out/
*.spec.bak
```

## 四、镜头表功能优化（如果保留）

### 16. 将镜头表功能从 copy 文件中拆出

- `gui_splitter_plus_copy.py` 已超过 1700 行。
- 建议拆成：
  - `shot_table_generator.py`
  - `shot_table_gui.py`
  - `llm_client.py`

### 17. API Key 支持环境变量

- 除了界面输入，也支持从环境变量读取 API Key。
- 示例：`OPENAI_API_KEY`。

### 18. 增强 LLM JSON 解析稳定性

- 当前 `_strip_code_fence` 只能处理简单 Markdown 代码块。
- 建议增加“从返回文本中提取第一个 JSON object”的逻辑。
- 避免模型返回少量解释文字时直接失败。

### 19. LLM 功能标记为可选

- 当前默认 Base URL 是 OpenAI 官方接口，容易让用户误以为工具必须联网。
- 建议界面明确说明：不启用 LLM 时使用本地模板生成。

### 20. CSV 字段支持模板配置

- 当前镜头表字段固定。
- 后续可支持不同平台模板：
  - 即梦图片提示词。
  - 可灵视频提示词。
  - 剪映备注。
  - Midjourney prompt。
  - Runway prompt。

## 五、建议的第一轮执行顺序

1. 将删除默认行为改为进回收站，并加强删除确认。
2. 新增 `requirements.txt`。
3. 新增 `.gitignore`。
4. 修复或归档旧版 `gui_splitter.py`。
5. 明确正式入口文件，并同步更新 `.spec`。
6. Worker 线程启动前快照 UI 配置。
7. 更新输出格式说明文案。
8. 决定是否正式保留镜头表功能；保留则合并，不保留则移入 `experimental/`。
